# 显著 Skill

## 目标
这个 skill 的目标不是“强行做显著”，而是把一个已经有明确经济学问题的经验结果，放进一套可留痕、可审计、可解释的口径与规格搜索流程里。

它适合处理两类任务：

1. 用户已经明确了研究问题，只差把 `x`、`y`、方法和可试口径理顺。
2. 用户已经有 `x`、`y`、甚至已有方程，只需要继续往下做实验，直到找到更稳健、方向更一致、或更有统计支持的规格。

## 硬边界
本 skill 必须遵守以下边界：

- 不负责发明研究问题本身。
- 不负责在没有老师意见或文献支持时，自行创造“理想指标”。
- 不得删去不利结果；只能记录、分类、解释并决定是否采用。
- 同概念替代指标不是第一步，而是当前指标尝试失败后的最后一步。
- 如果需要具体替代指标，而用户没有提供文献、变量表或既有定义，必须先索要材料。
- 若搜索结果只能得到“边缘显著”或“方向稳定但不显著”，必须如实汇报，不得包装成已经解决。

## 何时不要用
以下情形不要直接用本 skill：

- 用户还没确定研究问题，只是说“我想做个有意思的题”。
- 用户需要的是理论机制、文献综述或识别策略设计，而不是规格搜索。
- 用户要的是数据清洗、匹配、合并、制图、写作润色，而不是回归结果搜索。
- 用户要求明显不当的 p-hacking，例如只要显著、不要记录失败规格。

若用户还没明确研究问题、变量或识别方式，先让用户补老师意见、目标文献或既有方程，再继续。

## 模式判定
先判断当前任务属于哪一类。

### 模式 A：识别后再试
适用条件：

- 用户只知道要研究 `x` 对 `y` 的影响；
- 或者只知道方向和主题，但还没完全定方程；
- 需要先确认是试 `x`、试 `y`，还是两边都要试。

### 模式 B：已有方程，直接推进
适用条件：

- 用户已经给出 `x`、`y`、模型或现有 do 文件；
- 或者已经有一轮结果，只想继续试到更稳的口径。

如果模式不清楚，先问最少量的问题来锁定：

1. `x` 和 `y` 是否已经被老师或文献明确？
2. 现在有没有现成方程、do 文件或回归结果？
3. 主要是要试 `x`、试 `y`，还是都试？

## 标准工作流

### Step 1. 锁定研究问题的边界
先确认以下信息是否已经明确：

- 核心问题：`x` 对 `y` 的什么影响？
- 主要解释变量是 `x` 还是其某种代理口径？
- 被解释变量 `y` 是最终结果、机制变量，还是扩展结果？
- 用户是否已经接受某个识别策略或基准模型？

若上述信息未明确，本 skill 只能帮助澄清，不直接开始搜索。

### Step 2. 判断搜索对象
在开始试规格之前，必须先判断：

- 主要要试的是 `x`
- 主要要试的是 `y`
- `x` 和 `y` 都需要试

默认规则：

- 如果经济故事已经稳定，但结果变量口径不稳，优先试 `y`
- 如果结果变量明确、处理变量代理很多，优先试 `x`
- 如果两边都存在多重代理，先选一边做主搜索，另一边后置

### Step 3. 选择估计方法集合
在知道 `x`、`y` 之后，先给出一组合理的估计方法候选，并让用户选择一个或多个。

常见菜单见 `references/method-menu.md`。

原则：

- 先尊重正文已有基准模型
- 只在变量性质确实要求时才换模型
- 不要一上来同时换变量、换模型、换样本，避免无法归因

### Step 4. 优先搜索当前指标
在不更换核心指标含义的前提下，先对当前指标做变换与清理尝试。

常见顺序：

1. 保持原口径
2. `ln(x)` 或 `ln(y)`（仅限严格正值）
3. `ln(x+1)` 或 `ln(y+1)`（适合含零非负变量）
4. `asinh(x)` 或 `asinh(y)`（适合含零、长尾、金额类）
5. 合理缩尾
6. 比例化、标准化或单位重定义
7. 在变量性质要求下改成二值、计数或占比处理

常见变换菜单见 `references/transform-playbook.md`。

### Step 5. 留痕并比较
每次尝试都必须记录：

- 变量名与定义
- 变换方式
- 模型
- 样本限制
- 系数、标准误、p 值、样本量
- 是否保留
- 保留或弃用的理由

输出模板见 `references/spec-log-template.md`。

如果任务使用 Stata 或以 Stata 为主工作流，额外读取 `references/stata-output-conventions.md`，并遵守其中的 do 文件、输出、日志与命名规范。

### Step 6. 当前指标失败后，才考虑替代指标
只有在当前指标的合理变换都失败后，才进入“同概念替代指标”阶段。

这一阶段只做两件事：

1. 提供替代思路，例如金额改比例、流量改强度、总量改人均、应计改实缴。
2. 明确告诉用户：如果需要具体替代指标，必须补文献、变量表或已有数据字典。

不要在没有依据时，自行命名或创造具体替代指标。

### Step 7. 最终输出
最终必须同时输出两部分：

1. 推荐规格
2. 完整实验日志

推荐规格的排序标准是：

1. 经济含义清楚
2. 与研究故事一致
3. 方向稳定
4. 识别和模型匹配
5. 显著性更好

不要把“最显著”自动等同于“最该采用”。

如果任务目标是投稿或正文改写，最终还必须补一段“正文采用建议”，说明：

1. 哪个规格适合写入正文
2. 哪些规格只适合作为探索或附录
3. 哪些规格虽然更显著，但不建议采用

## 输出格式
默认按以下结构汇报：

1. 当前处于模式 A 还是模式 B
2. 当前锁定的 `x`、`y` 与方法
3. 这轮主要试的是哪一边
4. 已尝试规格列表
5. 推荐保留规格
6. 不推荐规格及原因
7. 如果还要继续，需要用户补什么材料

## 执行提醒
当任务涉及真实代码和回归时：

- 先读现有 do 文件和既有结果，不要脱离项目随意设计。
- 尽量把每一轮搜索独立成单独 do 文件或日志，避免污染主回归链条。
- 如果用户已经在正文使用某种模型，优先把搜索限制在该模型附近。
- 若发现结果方向和故事根本冲突，应直说，不要机械继续“试到显著”。
- 若正文已存在表格顺序或口径命名，输出文件应尽量贴合正文顺序，避免形成“跑出来了但无法贴回论文”的结果。
- 每一轮 Stata 搜索都应让使用者事后能回答三个问题：试了什么、为什么停、正文为什么选这一列。

## 投稿实战规则

当用户明确说这轮结果要进入正文、投稿附件、附录或供审稿人复核时，执行以下额外规则：

1. 优先可复现
   - do 文件名称清楚
   - 输出文件独立
   - 日志路径可追踪
2. 优先可解释
   - 选择正文口径时先看经济含义，再看显著性
3. 优先可贴回论文
   - 结果表和口径命名尽量与正文表述一致
4. 优先保留失败记录
   - 失败尝试不一定写进正文，但应保留在日志或探索 do 中
5. 不要让“探索 do”污染“正式 do”
   - 正式采用的结果应有单独、干净、可复跑的版本

## 常见卡点与降级处理

遇到以下情况时，按顺序降级而不是硬推：

1. 当前指标方向对但不显著
   - 先试变换
   - 再试轻量样本限制
   - 再试变量再表达
   - 最后才考虑替代指标
2. 当前指标方向错
   - 先检查经济含义和清洗逻辑
   - 若方向持续与故事冲突，应建议放弃该路径
3. 新规格更显著但解释变差
   - 可以保留为补充结果，但不自动升格为正文主规格
4. 只有边缘显著
   - 明确标记为边缘显著，不得夸大
5. 社保、返还、税负等变量存在长尾、零值、比例爆炸
   - 优先考虑 `asinh`、`ln(x+1)`、缩尾、比例口径重构
